
## Code Review: `UserApiClient.java` и `BaseTest.java`

## 1. Список найденных проблем

### UserApiClient реализован как singleton, хотя зависит от baseUrl (major)
```java
public static UserApiClient getInstance(String baseUrl)
```

**Почему это проблема:**
- baseUrl передается параметром, но фактически учитывается только при первом создании клиента. Все следующие 
вызовы будут возвращать уже созданный экземпляр, даже если передан другой URL

**Что сломается:**
- Как только тесты начнут использовать больше одного варианта параметра baseUrl - клиент продолжит ходить в 
изначальный baseUrl и тесты упадут, т.е. данная реализация делает невозможным:
  - запуск тестов против на несколько окружений
  - запуск части тестов с другим baseUrl

---

### Singleton не потокобезопасен (major)

```java
if (instance == null) {
    instance = new UserApiClient(baseUrl);
}
```

**Почему это проблема:**
- Метод `getInstance()` не синхронизирован. В параллельном запуске несколько потоков могут одновременно увидеть
`instance == null` и создать несколько экземпляров объекта

**Что сломается:**
- При параллельном запуске тестов поведение будет недетерменированным - несколько потоков могут одновременно получить instance == null,
создать несколько экземпляров new UserApiClient и статическое поле instance будет перезаписано неконтролируемым образом (будет использован
последний созданный объект). Если используется неодинаковый baseUrl - одинаковый набор тестов будет то проходить, то падать

---


### `BaseTest` смешивает UI и API слои тестирования (major)

```java
protected WebDriver driver;
protected UserApiClient apiClient;
```

**Почему это проблема:**
- Один базовый класс сразу отвечает и за браузер, и за API-клиент, хотя не всем тестам нужны
  обе зависимости

**Что сломается:**
- API тесты будут производить ненужный UI setup
- UI тесты будут производить ненужный API setup
- по мере роста количества настроек базовый класс разрастется и станет тяжелым в поддержке

---

### Возможен конфликт ответственности между SeleniumExtension и ручным созданием ChromeDriver (major)

- `@ExtendWith(SeleniumExtension.class)`
- ручной setup в `setUp()`

**Почему это проблема:**
- Из кода неясно, что именно делает SeleniumExtension. Если extension уже управляет
  настройкой вебдрайвера, то BaseTest нарушает принцип единой ответственности

**Что сломается:**
- Возможен дублирующийся менеджмент настроек вебрайвера;
- Тесты и extension могут начать работать с разными экземплярами драйвера.

---


### BaseTest привязан к локальному окружению (major)

```java
System.setProperty("webdriver.chrome.driver", "/usr/local/bin/chromedriver");
apiClient = UserApiClient.getInstance("http://localhost:8080");
```

**Почему это проблема:**
- Путь к драйверу и `baseUrl` захардкожены прямо в базовом тестовом классе

**Что сломается:**
- Тесты не запустятся на машине, где драйвер лежит в другом месте
- Для удаленных запусков придется менять код вместо конфигурации

---

### Метод deleteUser() скрывает ответ сервера (major)

```java
public void deleteUser(String userId)
```

**Почему это проблема:**
- Метод возвращает void, поэтому вызывающий код (если проверки вынесены в хелпер) не может
  проверить статус-код, статус-лайн, тело ошибки или факт успешного удаления

**Что сломается:**
- Тест продолжит выполнение даже если API вернул ошибку
- Ошибка всплывет позже в другом месте, что затруднит разбор

---

### В API-клиенте нет общей конфигурации запроса (major)

```java
все методы UserApiClient
```

**Почему это проблема:**
- baseUri, contentType и в будущем другие общие настройки задаются вручную в каждом методе.
  Это уже дублирование, а по мере роста клиента в нем появятся общие заголовки, auth, логирование,
  таймауты и фильтры

**Что сломается:**
- Один метод обновят, другой забудут
- API-клиент начнет вести себя неравномерно
- Поддержка станет дороже, потому что любое общее изменение придется вносить во многие методы

---

###  В API-клиенте нет базового логирования запросов и ответов (major)
```java
все методы UserApiClient
```

**Почему это проблема:**
- При падении теста нет гарантии, что в аттачах окажутся нужные данные и таймштампы. Это усложняет
  расследование падений

**Что сломается:**
- Ошибки от апи будут расследоваться дольше
- Плавающие баги будет сложнее локализовать

**Нюанс:**
Логирование нужно делать аккуратно: чувствительные данные должны маскироваться

---


### Глобальный implicitWait может ухудшить предсказуемость UI-тестов  (major)

```java
driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
```

**Почему это проблема:**
- implicitWait глобально влияет на поиск элементов и может замедлять прогоны. Как более устойчивый подход лучше
  использовать явные ожидания и Selenide

**Что сломается:**
- Ошибки поиска элементов будут проявляться медленнее
- Из теста будет неочевидно, какого именно состояния элемента мы ждём

---


### tearDown() может упасть с NullPointerException (major)

```java
driver.quit();
```

**Почему это проблема:**
- Если инициализация драйвера упадет в setUp(), поле driver останется null, и cleanup завершится
  дополнительной ошибкой

**Что сломается:**
- Исходная причина падения будет замаскирована NullPointerException в tearDown()
- Логи станут менее информативными для расследования

---


### Поле timeout не используется (minor)
```java
private int timeout = 5000;
```

**Почему это проблема:**
- Поле создает ложное впечатление, что у клиента есть настроенные таймауты, хотя фактически это
  значение не применяется

**Что сломается:**
- Если таймауты планировалось реализовать - при подвисании ответа от сервиса клиент не будет
  приостанавливать работу по таймауту

---

### В deleteUser() userId вставляется в URL строковой конкатенацией (minor)

```java
.delete("/api/v1/users/" + userId);
```

**Почему это проблема:**
- Это менее безопасно и менее единообразно, чем использование `pathParam`, которое уже 
применяется в getUserById()

**Что сломается:**
- Если userId содержит спецсимволы, пробелы или слэши, запрос может уйти не по тому пути
- Поведение одинаковых по смыслу методов будет различаться

---

### getUserById() лучше переименовать в readUser() для консистентного CRUD-именования (minor)

```java
public Response getUserById(String userId)
```

**Почему это проблема:**
Если клиент оформляется как CRUD-обертка над сущностью User, именование лучше держать единым:
create / read / update / delete. Текущее имя технически понятно, но выбивается из общего стиля

**Что сломается:**
- Ничего критичного не сломается, но по мере роста клиента появится смешение стилей:
get, find, read.

---

### В каждом тесте создается новый экземпляр вебдрайвера (minor)

```java
@BeforeEach @AfterEach
```

**Почему это проблема:**
- Поднимать новый браузер на каждый тест просто и изолированно, но при большом наборе UI-тестов
это нагружает инфраструктуру и замедляет прогоны

**Что сломается:**
- Время прогона будет расти за счет старта браузера, а не за счет самих проверок
- CI будет масштабироваться хуже

---


## 2. Вопросы на обсуждение с командой

### Где должна проходить граница между транспортным и валидационным слоем проекта

Должен ли UserApiClient только отправлять запросы и возвращать Response, или он должен сразу валидировать expected status и response contract

**Почему это важно обсудить:**

Это архитектурное решение должно быть единым для всех клиентов. Иначе получится хаос:

- один клиент только отправляет запросы
- второй валидирует статусы внутри
- третий частично скрывает ошибки

**Нужно определить:**

- единый контракт для API clients
- где живут ResponseSpecification
- где делать бизнес-валидацию
- что возвращать наружу, Response, DTO, typed result или assertion wrapper

---

### Кто должен управлять жизненным циклом WebDriver: SeleniumExtension или BaseTest

SeleniumExtension или BaseTest

**Почему это важно обсудить:**

Два источника правды недопустимы. Если extension уже отвечает за настройку драйвера, то BaseTest должен только использовать его, а не создавать свой экземпляр.

**Нужно определить:**

- единый владелец инициализации
- единый владелец очистки
- где живет конфигурация драйвера (параметры, таймауты, логирование)
- как эта схема поддерживает параллельные запуски и расширяемость

---

### Стратегия тестовых базовых классов и фикстур

Нужен ли вообще единый BaseTest, или стоит разделить инфраструктуру на UiTestBase, ApiTestBase, extension-based или factory-based composition

**Почему это важно обсудить:**

Текущий подход удобен на короткой дистанции и разрушителен на длинной. Он провоцирует рост общего базового класса, в который начинают складывать все подряд

**Нужно определить:**

- использование наследования или композиции
- как избежать god-классов 
- как разделять зависимости по типам тестов

---

### Стратегия конфигурации окружения

Как реализуем - через properties, environment variables, profiles или config objects

**Почему это важно обсудить:**

Пока конфиг зашит в код, фреймворк не готов ни к CI, ни к multi-env, ни к удобной локальной разработке

### Переиспользовать вебдрайвер между тестами

Создавать новый браузер на каждый тест проще и надежнее с точки зрения изоляции. Переиспользование браузера ускорит тесты, 
но только если команда готова принимать во внимание этот нюанс, поддерживать и использовать.

**Почему это важно обсудить:**

Нужно принять общее решение - что важнее для проекта: скорость или максимальная изоляция, на каком 
уровне переиспользовать драйвер, как полностью очищать состояние.                                                                                                                                                                                                                                                                                                                                                                                                                       

---

## 3. Предложения по рефакторингу

- Убрать singleton-паттерн из UserApiClient и создавать клиент как обычный объект
- Вынести общую конфигурацию Rest Assured в RequestSpecification
- Добавить в UserApiClient логирование запросов и ответов
- Определить единое правило: где именно валидируются ошибки API-ответов - в клиенте или в helper слое 
и вынести это в конфигурацию ResponseSpecification
- Возвращать Response в deleteUser() либо валидировать ожидаемый статус внутри метода
- Использовать pathParam() вместо строковой конкатенации в URL
- Переименовать getUserById() в readUser()
- Вынести baseUrl, путь к драйверу и другие параметры окружения в конфиг
- Перенести setup браузера в SeleniumExtension, если extension уже отвечает за это
- Обсудить стратегию переиспользования WebDriver для ускорения прогонов
- Заменить глобальный implicitWait на  explicit waits
- Добавить безопасный cleanup: if (driver != null) driver.quit();
- Разделить базовые классы по тест сьютам, сетап для UI и API вынести в аннотации: @UiTest и @ApiTest и аннотировать
нужные базовые классы, в зависимости от того, что тестируют сьюты. Это позволит избежать ненужного setup для тестов, которым не нужны обе зависимости.
